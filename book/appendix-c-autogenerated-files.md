# Appendix C ‚Äî Files CMake Generates for You

A recurring question in this course was "which files are autogenerated?" Here's
the complete picture. **Rule of thumb: everything under `build/` is generated
and must never be committed or hand-edited.** That's why the very first commit
included a `.gitignore` with `/build/`.

## Everything in `build/`

| Path | What it is | Who makes it |
|------|-----------|--------------|
| `build/CMakeCache.txt` | Cached settings: compiler path, generator, every `option()`/`-D` value | `cmake` configure |
| `build/CMakeFiles/` | Internal bookkeeping ‚Äî compiler checks, dependency data | `cmake` configure |
| `build/build.ninja`, `build/rules.ninja` | The Ninja build description CMake wrote | `cmake` configure (Ninja generator) |
| `build/cmake_install.cmake` | Install script | `cmake` configure |
| `build/compile_commands.json` | Exact compile flags per source file (for IntelliSense) ‚Äî only when `CMAKE_EXPORT_COMPILE_COMMANDS` is ON | `cmake` configure |
| `build/generated/app_config.h` | Config header filled in from `config/app_config.h.in` | `configure_file()` |
| `build/hello.exe` | The compiled app | the build (Ninja) |
| `build/**/lib*.a` | Static libraries (`libmenus.a`, `libadc.a`, `libmath_utils.a`) | the build |
| `build/tests/test_*.exe` | Compiled test programs | the build |
| `build/tests/cmocka.dll` | Copied next to the tests by a POST_BUILD step so they run | the build |

With the **Visual Studio** generator instead of Ninja, `build/` would hold
`*.sln` and `*.vcxproj` files (and the exe under `build/Debug/`) ‚Äî same
`CMakeLists.txt`, different generated files. That's the generator's job
(Chapter 5).

## The two "template ‚Üí generated" pairs you wrote

Some generated files come from a template *you* maintain. You edit the template;
CMake writes the output into `build/`:

| You edit (committed) | CMake generates (in build/, ignored) | Via |
|----------------------|----------------------------------------|-----|
| `config/app_config.h.in` | `build/generated/app_config.h` | `configure_file()` |
| ‚Äî | `build/compile_commands.json` | `set(CMAKE_EXPORT_COMPILE_COMMANDS ON)` |

Never edit the generated `app_config.h` ‚Äî your changes vanish on the next
configure. Change the `.in` template or the CMake `option()`/`set()` values
instead.

## Regeneration timing

- **Configure-time** artifacts (cache, `build.ninja`, `compile_commands.json`,
  `app_config.h`) are (re)written when you run `cmake -S . -B build ...`.
  Changing an `option()` value or a template requires re-running that step ‚Äî a
  plain `cmake --build build` will not regenerate them.
- **Build-time** artifacts (`.exe`, `.a`) are produced by `cmake --build build`
  and rebuilt incrementally when their sources change.

## To start completely fresh

```bash
rm -rf build
```

Nothing of value is lost ‚Äî it all regenerates from your committed sources.
